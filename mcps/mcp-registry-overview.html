The Problem

What's Missing

There is no single place that tracks what MCP servers exist, what state they are in, which are intended for consumption, or how a deployed instance relates to the asset it was deployed from. The MCP Registry fills this gap by becoming the governed record that other components read from.

The Registry

What the MCP Registry Does

The registry stores metadata-first records for MCP servers. Each record carries the fields below. Downstream components use publish state to decide what to surface.

Publish State Replaces "Approval"

For 3.5 Dev Preview, we use publish state instead of "approval." No workflow engine or policy scaffolding required — just a concrete lifecycle position that controls downstream surfacing.

Draft
→
Published
→
Deprecated
identity

Identity & Version

Stable name, version string, descriptive metadata. Each MCP server is a governed asset with versioned records.

server_json

MCP Payload

The canonical server definition aligned with the upstream MCP registry schema. Immutable per version.

publish_state

Publish State

Draft, published, or deprecated. Only published MCPs are eligible for downstream surfacing to catalog, gateway, and Studio.

source

Source & Hosting

Whether the MCP is internally or externally hosted. Both use the same normalized record shape.

deployment_ref

Deployment References

Optional association with a deployed instance — endpoint, deployment details. Registry does not deploy; it records the association.

workspace

Workspace Scope

Visibility boundary for RBAC-aware access. Users and integrations see only records within their scope.

ⓘ
Mandatory vs. Optional Metadata

Identity fields (name, version, publish state) are mandatory on every record. Extended metadata (description, tags, labels, deployment references) is optional and can be populated incrementally as the record matures.

≠
What the registry does not do

Host MCP runtimes, enforce gateway policies, deploy services, or replace the catalog. It stores the governed state that those components consume.

3.5 Dev Preview Scope

Targeted User Stories

Eleven stories defining the minimum user-visible and system-visible behaviors for the first MCP Registry deliverable. Each story describes who acts, what changes, and which component interaction it implies.

01
Platform Engineer

Register an internal MCP server

Create a governed record with stable identity, version, and metadata. The record starts in draft publish state.

  • Metadata-first reference, not the runtime itself
  • Canonical MCP payload aligned with upstream schema
Core registry write surface. No downstream interaction until published.
02
Platform Engineer

Register an external MCP server

Govern externally hosted MCPs through the same platform experience without requiring the registry to host the runtime.

  • Stores endpoint or source references
  • Same record shape as internal MCPs
  • Version defaults to "latest" when unspecified — declared, not enforced
Downstream components consume one normalized record regardless of hosting.
03
Platform Engineer / Integration

Retrieve and list MCP records

Retrieve by identifier and version, or list with structured field filtering. Results include identity, version, metadata, publish state, and references.

  • Filter by name, publish state, workspace scope, hosting model, and version
  • Results filtered by caller's access scope
Core read surface for catalog, gateway, and AAA/Studio.
04
Platform Engineer

Change publish state

Move an MCP between draft, published, and deprecated. Only published MCPs are eligible for downstream surfacing.

  • Deprecated remains queryable, not preferred for new use
  • Transitions recorded for auditability
Replaces approval workflows for 3.5 DP. Components filter on this state.
05
Catalog Integration

Surface publishable MCPs to the catalog

Catalog reads only published, scope-appropriate MCPs from the registry for discovery and selection.

  • Catalog does not define its own governance state
  • Catalog sync queries published + deprecated (multi-state) so users see deprecation warnings
Registry = governance truth. Catalog = discovery surface.
06
Platform Engineer

Associate deployment or runtime references

Link a deployed MCP instance to its governed registry record so deployment is not disconnected from governance.

  • Registry does not deploy — lifecycle operator does
  • In 3.5 DP, associations may be written manually
Bridges "what is governed" and "what is running."
07
MCP Gateway Integration

Provide gateway-relevant MCP metadata

Gateway reads identity, version, endpoint references, and publish state from the registry for runtime mediation.

  • Gateway remains the runtime control layer
  • Registry stores the deployment endpoint (from lifecycle operator); gateway creates its own mediated route
  • Full automation is a later-phase goal
Registry becomes governed source for gateway metadata.
08
AAA / Studio Integration

Surface consumable MCPs to AAA / Studio

AAA/Studio retrieves published, scope-filtered MCPs so AI engineers see only what is intended for consumption.

  • Governed alternative to config-map-driven exposure
Shift from static config to registry-governed consumption.
09
Platform Engineer / AI Engineer

Enforce scoped visibility

Users see only MCP records within their workspace scope. Applies to both management and consumption queries.

  • Workspace-based visibility for MVP
  • Does not require a policy engine
All downstream surfaces receive scope-filtered views.
10
Platform Engineer

Preserve version history and deprecate older MCPs

Retain prior versions, each with its own publish state. Mark older versions deprecated without erasing history.

  • Downstream consumers see deprecation status
  • Supports auditability without full audit logging
Catalog and Studio guide users toward current versions.
11
Platform Engineer

Register an already-deployed MCP server

Bring an MCP server that was deployed outside the platform (e.g. from a catalog browse-and-deploy flow) under registry governance after the fact.

  • Lifecycle operator writes a deployment reference back to the registry after deployment
  • Registry creates a new record (draft) linked to that deployment reference
  • Platform engineer can then publish the record to make it visible to catalog, gateway, and Studio
  • Ensures no deployed MCP server remains ungoverned
Closes the "catalog → deploy → never registered" gap. Lifecycle Operator → Registry write-back.

Boundaries

3.5 Dev Preview Scope

For 3.5 DP, the registry stores enough state for components to read from it. Full automation of every handoff is a later-phase goal.

In Scope

  • Register MCP servers (internal and external) as governed, versioned records
  • Metadata-first records with canonical MCP payload
  • Publish state (draft / published / deprecated) controls downstream surfacing
  • Read access for catalog, gateway, and AAA/Studio filtered by publish state and workspace
  • Associate deployment or runtime references with governed records
  • Version history and deprecation
  • RBAC-aware scoped visibility on all operations

Out of Scope

  • Approval workflows, review gates, or policy engines
  • Certification or verification status tracking
  • Full automated lifecycle handoffs between components
  • Detailed API, CRD, or data model design
  • Full multi-asset registry framework
  • Disconnected sync/mirroring
  • UI/UX design
