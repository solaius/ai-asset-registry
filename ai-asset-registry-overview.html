The Problem

AI is Expanding Beyond Models

Organizations need to govern agents, MCP servers, prompts, skills, and more. The current model registry was not purpose-built for this — it was built for models and extended. Building a bespoke registry for each new asset type does not scale.

The Reality Today

In RHOAI 3.4, an MCP server goes from catalog to consumption through 5 steps across 4 disconnected systems, with 2 manual hand-offs and zero governance checkpoints. No system records who approved it, what version is deployed, or whether it passed security scanning. Now imagine doing this for agents, skills, prompts, guardrails, and every future AI asset type.

Fragmented Governance

Each new AI asset type would need its own registry effort. Duplicated work, inconsistent UX, slower time-to-market for new asset support.

Manual Stitching

Today, connecting catalog → deployment → gateway → Studio requires manual steps at every stage. Fragile and ungoverned.

No Source of Truth

No single system knows which assets are approved, deployed, consumable, or how they relate to each other.

Agent Sprawl

Enterprises face uncontrolled proliferation of agents, models, and tools. Uncoordinated proliferation creates risk, not value.

The Solution

Three Parts, One Framework

The AI Asset Registry spans three distinct areas. The plugin architecture is what we are proposing now. Gen AI Studio already ships in RHOAI 3.4. The ecosystem is what comes next.

What "Plugin Architecture" Means Concretely

The plugin model extends MLflow's artifact type system so each AI asset type registers as a typed plugin with its own metadata schema, lifecycle states, and governance rules — without forking or rebuilding the core registry. One framework, many asset types, zero bespoke registries.
